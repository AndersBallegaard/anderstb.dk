A Glimpse into the Future: An introduction to IPv6 in your homelab

Anders Ballegaard | Jul 5, 2025 min read

Introduction

I have been a big supporter of ipv6 for a long time. IPv6 has been a big part of my homelab for years, and as part of my work at a major Danish ISP, I have also helped enabling and improving ipv6 for many Danish broadband customers.

I am currently working on updating and fixing some things in my homelab, and I thought i would write about it here. Partly as documentation for myself, but also as an introduction for anyone curious about how to start on thier own ipv6 homelab journey.

But why should you care about ipv6? well let’s look at how ipv6 is used today.

First of all nearly half of all internet traffic is ipv6, the numbers varrys a bit but here are the numbers from Google and Meta as of writing

Source Source

Besides the fact that a large portion of the internet is already using IPv6, there are also pushes from both companies and goverments to move to ipv6. Some of those major pushes include:

  • Apple requires all app store apps to support working in ipv6 only networks. They have required this since 2016.
  • Sevral mobile operators have have deployed ipv6 only mobile networks with 464XLAT being the only way of accessing ipv4 sites. In the west the most notable example is probably T-Mobile in the US. But to my knowlage this aproach is also common in developing countries due to ipv4 scarcity.
  • US Office of Management and Budget ipv6 mandate. In 2023 the US federal government presented a quite ambisious plan for moving to ipv6.
  • China have mandated that Chinese router manufacturers must enable ipv6 by default in all new routers they sell.
  • Most major cloud providers have started not including public ipv4 addresses for free, thus adding an extra cost for still running ipv4 to the server directly. While this doesn’t force organizations to change, it is a nudge that can be used as a motivator.

Ofcourse companies and goverments isn’t just pusing for ipv6 for no reason at all. It takes a lot of effort to change, so there needs to be some good reasons behind the change. So here are some of the reasons:

  • We are running out of IPv4 Address space. Part of this problem is related to the fact that early ipv4 allocation was made in a very short sighted way, but unfortunately there is not really a way to change this. (and no Class E or redefining 127.0.0.0/8 won’t work). Unlike a lot of ipv6 supporters, I don’t like to say we have run out, but instead say we are running out. While it is right that getting new ipv4 space directly from your RIR is imposible (or close to it), there is still healthy resale market. So you CAN get ipv4 space, but supply and demand makes a pure ipv4 only internet an impossibility now due to the amount of things we want connected.
  • Simpler routing, and simpler network operations. This might sound counterintuitive for anyone who have grown up with ipv4 networks, and I do admit, it takes some time getting used to. But once you see the beuty in always using /64 netmasks, without having to worry about exsation, or when you start to apreciate the simplicity of not dealing with NAT when troubleshooting, or realize the simplicity of the (base) ipv6 header compared to ipv6. Like all things there is a learning curve, and the more time you have spent with ipv4 the harder it probably is, but the more you use ipv6, the easier it becomes, and the more you will love it.
  • Decreased latency. I will admit this is not a universal truth, but removing NAT on the internet, does decrease latency, especially if your ISP is forcing you though CGNAT routers placed outside the optimal network path. In some cases we also see a decreased latency due to cutting out legacy infrastructure that only support IPv4.
  • Energy efficiency. Kinda the same as latency, removing NAT removes compute cycles to do NAT, and decreases power consumption.
  • Protocol improvements. Ok now we are getting nerdy, and some of these have not been realized to the point they should be yet, but here are some of my favorite protocol improvements:
    • Extention headers. This is the feature enabling most of the protocol improvements, but on older hardware it is also a source of fustration.
    • Routing header - This allows the source device to specify the path it wants to take through the network. A very cool application of this is SRv6 routing.
    • IPSEC header - This allows for encryption and authentication of packets built directly into the IP protocol, instead of as an additional layer like it is in IPv4.

So as I hope you agree, no matter if you are building networks, or building the apps on top of the networks, you need to understand how to work with ipv6. I belive experimentation is the best way to learn about a new technology, this is why i would highly encourage you to include ipv6 in your labs.

If it sounds interesting then keep reading this, and future blog posts where we are diving into the details of some of these topics.

IPv6 Mostly vs IPv6 only

It’s probably important to start out defining what i am trying to achive, and what some common terms mean.

IPv6 Only

This is pretty staight forward, it means that you ONLY have access to an ipv6 network. Unless you understand your devices and applications very well, this is probably not a good idea right now.

IPv6 only is ofcourse the ultimate goal, but we aren’t there yet. So instead most networks are targeting ipv6 mostly as a stepping stone.

Ipv4 connectivity might still be provided for backwards compatibility though NAT64.

IPv6 Mostly

This is accually a defined term, if you want the full version see IETF draft-ietf-v6ops-6mops-01 But here is the short version:

  • The network must work for IPv6 only clients, dual stack clients, and IPv4 only clients. The goal being to provide a space for migrating clients towards ipv6 only.
  • The network must provide a NAT64 solution to the clients, notably there is no requirement for providing a DNS64 solution.
  • The networks DHCPv4 server(s) must include DHCP option 108 in responses to clients. This indicates to hosts that support ipv6 only, that the network also supports ipv6 only. Option 108 essentially let’s a device skip getting an IPv4 address.

So what am i choosing?

My target for now is IPv6 Mostly, for the following reasons:

  • I own devices that doesn’t support ipv6, or atleast doesn’t support ipv6 only operations.
  • This is the most common deployment method
  • It doesn’t limit me from running some devices as ipv6 only for testing purposes.

So how do i access ipv4 only sites?

The short answer is NAT64 + either DNS64 or CLAT.

I will dedicate a blog post in the future to NAT64, but here is the short version of what it does. Due to IPv6 having more bits than IPv4, we can cram an ipv4 address into an ipv6 address. We traditionally use 64:ff9b::/96 for this, but there are other options. So let’s say you wanted to access 1.1.1.1 via NAT64, instead of sending your packet to 1.1.1.1, you would send it to 64:ff9b::101:101 given that is what the address would be if you took the first 96 bits from 64:ff9b:: and added the 32 bits of 1.1.1.1.

But we are (mostly) not accessing services directly by ipv4 address, so we need to map DNS to this mess, somehow. There are two ways this is done

  • DNS64 - This is essentially the DNS server lying to the client, by creating a fake AAAA record though the NAT64 device if no AAAAs exist for that domain. But given the DNS server is lying to the client, DNSSEC doesn’t like DNS64. The advantage is that it works on any device that supports IPv6. But it only works for DNS, so any IPv4 littrals won’t be saved by this. Another indirect consequence of this approach is that sites with AAAA records, but broken ipv6 doesn’t have any way to fall back to the ipv4 connectivity.
  • CLAT aka 464XLAT - This works by having code on the device doing the translation, it’s typically implemented as a new ip on an existing interface, or new interface entirely. This is very common in mobile devices, and it is (very slowly) getting implemented on desktop devices. The advantage is that this works for both DNS and IPv4 literals, and it doesn’t involve changing DNS responses.

A short introduction to my home network, and what i want to do.

To say that my home network is weird would be an understatement. Like a lot of people working in IT, i have a sizable homelab, but unlike most others, i have decided to somewhat seperate my lab from the rest of the network. Oh and then there is the small detail that i am running my own publicly routed ASN (AS201911), and though that have a /44 ipv6 allocation.

The following is a diagram from earlier this year, of how i wanted the network to look logically. Some of this isn’t implemented, but it gives a picture of the direction i have been going

I will fully acknowledge that best practice is a unknown concept in this rats nest of a network. But my goals have never been to create something that made sense, it’s been to create something that gave me the flexibility i wanted to do whatever i want with limited impact on other parts of the network. Besides that i just like BGP, and wanted more BGP in my home network.

I don’t have a public IPv4 address for my home network, so everything i expose is exposed though ipv6 only, mostly with cloudflare proxy in front of the service, both to protect the service, and to enable dual stack access though cloudflares proxy service.

All routers you see in the diagram is either OpnSense firewalls or VYOS routers.

So what do i want to do with the network?

  • Create a centralized NAT64 service. Right now the DKNIM-LFW cluster, and DKNIM-HFW clusters are both running NAT64, i would like to centralize this.
  • Enable option 108 on all networks with DHCP. A lot should already have it, but it’s not enabled everywhere.
  • Explore running CLAT on linux servers.
  • Explore options for a perment ipv6 only or dual stacked container platform.

Expected challenges

Having been on this journey before, i don’t expect everything to be smooth sailing. I kinda know what to expect, but if you are starting an IPv6 mostly journey, here are some things to be aware of.

  • There are a few popular services using ipv4 littrals, most notably Discord. So if you enable option 108 on a device without CLAT, don’t be surpriced when parts of Discord stops working.
  • You might find that your ISP doesn’t support ipv6, you can ofcourse solve this in the crazy person way and start your own ISP like network, or you could be more sensible, and use something like HE tunnels.
  • IOT devices generally don’t have great ipv6 support.
  • If you are used to doing music streaming from your phone to maybe a Sonos speaker, that might break with option 108, given that Sonos doesn’t support ipv6, and your phone most likely won’t have an ipv4 address.
  • Containers and ipv6 - Generally not a good time.
  • Some applications you host might listen to 0.0.0.0 instead of [::] (this supports both v4 and v6), if it’s an open source project, and you have the ability, please fix it in the project, and try to get it merged.

What is next?

My plan is to start looking into diffrent NAT64 options given i have been out of that game for a bit. So look forward to a post comparing different options, and detailing what i will end up doing.
